## OVERVIEW

Database package for PostgreSQL access through Drizzle ORM, with schema-first types and insert validation.

## WHERE TO LOOK

- `src/index.ts` for the public package surface, including `pool`, `db`, and schema re-exports.
- `src/schema/index.ts` for the schema barrel used by both runtime code and the `./schema` subpath export.
- `src/schema/conversations.ts` for the basic table pattern used in this package.
- `src/schema/messages.ts` for a schema file that adds insert-shape omissions on generated fields.
- `src/schema/generated-images.ts` for a schema file with the same inferred-type pattern applied to another table.
- `package.json` for the package exports and database push commands.
- `drizzle.config.ts` for Drizzle Kit configuration.
- `dist/` only to confirm declaration output shape, never as an editing target.

## CONVENTIONS

- Treat `src/schema/*` as the source of truth. Keep schema definitions, insert validation, and exported inferred types together in the same file.
- For each table, prefer this flow: define the table, create the insert schema with `createInsertSchema`, then export read and write types.
- Export read models from Drizzle using `typeof table.$inferSelect`.
- Export insert models from Zod using `z.infer<typeof insertSchema>`.
- When a column is generated by the database or application, omit it from the insert schema instead of widening the insert type manually.
- Keep `src/schema/index.ts` updated whenever a new table file is added, because consumers may import through both the package root and `@workspace/db/schema`.
- Preserve the public surface in `src/index.ts`: package consumers expect `pool`, `db`, and schema exports there.
- This package builds as declaration-only output. Avoid changes that depend on emitted runtime files from `dist/`.
- Keep naming explicit and paired, for example `messages`, `insertMessageSchema`, `Message`, `InsertMessage`.
- If you add a table, make its schema and exported types match the surrounding files before touching downstream packages.
- Use `npm run -w @workspace/db push` for normal schema pushes.
- Use `npm run -w @workspace/db push-force` only when the task explicitly requires a force push.

## ANTI-PATTERNS

- Don't hand-write insert payload types when `createInsertSchema` already defines the contract.
- Don't export table internals from ad hoc files outside `src/schema/`; keep database shapes discoverable in the schema directory.
- Don't bypass `src/schema/index.ts` when adding new schema modules, or the `./schema` export will drift.
- Don't edit files under `dist/`; they are build artifacts.
- Don't collapse read and insert types into one type. The package relies on separate database row and input contracts.
- Don't add loose validation around inserts when the schema file can express the rule directly.
- Don't introduce database commands or migration workflows here that differ from the existing `push` and `push-force` entry points without a strong reason.
